1. Overview & architecture

The diagnostic is a standalone Qt 6 desktop application that reads and writes CiA 402 motor drives over a hal_can transport (UDP-multicast for loopback / bench work, ZLG USB-CANFD for real drives). It wraps the master_mgr + motor_drive_interface C layers from vr-mc-sdk in a dedicated Qt worker thread, surfaces live TPDO telemetry in a QtCharts scope, and exposes every commissioning knob through modeless dialogs that can sit next to the main window while the operator works.

The layering is intentional:

  • UI thread builds widgets, routes user input, paints charts. Never blocks on SDO.
  • Worker thread owns master_mgr_t, runs the periodic refresh, and performs all SDO read/write work. Emits a snapshot signal at 100 Hz.
  • SDK layer (C code) — master_mgr fans out to per-slave motor_drive_interface; the CiA 402 implementation talks to co_fd_usdo (SDO) and can_fd_pdo (PDO) over a hal_can vtable.
  • Transport layer — hal_can_udp or hal_can_zlg concrete, chosen at connect time.
Thread safety. Every exchange between the UI and the worker goes through Qt signals with Qt::QueuedConnection. No raw pointer handoffs, no shared-state races. Even the periodic refresh grabs the worker's mutex, so an Apply press and a refresh tick serialise naturally.

2. Install & first run

2.1 Prerequisites

  • Qt 6.2 or newer (Core, Gui, Widgets, Charts).
  • CMake 3.16+, a C++17 compiler.
  • Linux host — hal_can_udp uses BSD sockets + multicast; hal_can_zlg loads libcontrolcanfd.so.
  • For the ZLG path: a USB-CANFD adapter plugged in.

2.2 Build from source

git clone …/vr-hand
cd vr-hand/vr-mc-diagnostic
cmake -S . -B build
cmake --build build --target vr_mc_diagnostic -j
./build/vr_mc_diagnostic

2.3 Loopback bring-up (no hardware)

The SDK ships a CiA 402 simulator you can use to bring the diagnostic up without a real drive. Run it in one terminal, then launch the diagnostic in another: 

./vr-mc-sdk/build/app/motor_drive_cia402 --udp --node 5
./vr-mc-diagnostic/build/vr_mc_diagnostic --auto-connect

The simulator emits TPDO1 at 2 kHz on multicast 239.192.0.42:23400; --auto-connect wires the diagnostic to the same group and pulls up node id 5.

3. Connecting to drives

3.1 Opening the Connect dialog

From the menu: Connection → Connect…. From the keyboard: Ctrl+K. The dialog offers two transports:

TransportFieldsWhen to use
UDP multicast Group (239.192.0.42), Port (23400) Loopback against the motor_drive_cia402 simulator, multi-host bench.
ZLG USB-CANFD Channel index, arbitration bitrate, data-phase bitrate Real drives on a USB-CANFD adapter.

Both transports take First node id and Count. The diagnostic sequentially registers that range of slaves into master_mgr, each gaining a row in the table.

3.2 After Connect

  • Status bar flips to connected with the slave count.
  • Table fills: state = offline transitions to the live statusword-decoded string once TPDO1 starts flowing.
  • Reconnect (Ctrl+R) replays the last config without re-opening the dialog.
  • Disconnect tears everything down idempotently — worker thread stays alive for the next connect.

4. Main window tour

4.1 Toolbar groups

Left to right, the toolbar is grouped by intent:

  1. File — Open / Save motor profile.
  2. Connection — Connect… / Disconnect / Reconnect.
  3. View — toggle the Profile, Tuning, and PDO map docks. Same actions also live in the View menu.
  4. Drive — Configure Drive… (Ctrl+Shift+C) + Upload Firmware.
  5. Data — Record telemetry + Clear charts.
  6. Help — About.
  7. E-STOP — always on the right, always clickable, shortcut Space.

4.2 Slave table

One row per slave. Columns: idx · id · name · state · online · pos · vel · trq. Selecting a row routes it to every per-slave panel — Control, Gains, Signal gen, Telemetry, and the dialogs that need a target.

5. Left-tab panels

5.1 Control tab

Drives the CiA 402 state machine. Each button issues a controlword transition through master_mgr_*_one:

  • Bringup — walks the drive from Not Ready to Switch On to Switched On.
  • Enable — final transition into Operation Enabled. Only state that accepts motion commands.
  • Disable — back to Switched On, torque off.
  • Fault reset — pulses controlword bit 7 to clear a fault; drive lands in Switch On Disabled.

Mode selector picks none · torque · velocity · position; Setpoint sends the numeric target. Both route through master_mgr_set_target_one.

5.2 Gains tab

Three loops (current · velocity · position) with independent Kp and Ki entry. Reads round-trip through motor_drive_intf_get_gain; writes go through motor_drive_intf_set_gain.

5.3 Signal generator

  • Constant / Step — one-shot setpoint.
  • Ramp — slope = amplitude / duration.
  • Sine — pure tone at the given frequency.
  • Chirp — log-swept sine from frequency start → end across duration; feeds the Bode view.

6. Detachable panes

Profile, Tuning, and PDO map are created as QDockWidgets — hidden and floating by default so the main window stays uncluttered. Clicking their toolbar button (or the View menu entry) surfaces the dock; the operator can float or re-dock it to any edge of the main window.

6.1 Motor profile

Edit… opens a modal form editor. File → Save Profile persists to JSON.

6.2 Tuning

  • Step — reports overshoot (%), 10–90 % rise time, and steady-state error.
  • Bode — log-sweep chirp + Goertzel-correlation yields magnitude (dB) and phase (°) vs. frequency on a log axis.

6.3 PDO map

7. Configure Drive

Drive → Configure Drive… (Ctrl+Shift+C) opens a modeless tabbed dialog targeted at the currently-selected slave. The dialog kicks off an SDO read on open so the form reflects what's on the drive, not a blank default.

7.1 Homing tab

FieldODNotes
Method0x6098Combobox with standard CiA 402 methods (0, 1, 2, 7, 11, 17, 18, 23, 27, 33–35, 37, −1, −2).
Home offset0x607COrigin shift applied by the drive after homing completes.
Switch search speed0x6099:1Fast approach until a switch / limit edge is detected.
Zero search speed0x6099:2Slow creep onto the actual zero (index / hard stop).
Homing acceleration0x609ARamp to the search speeds.

Two commissioning one-shots live below the form:

  • Start Homing — writes mode = 6 to 0x6060, then pulses controlword bit 4 on 0x6040 (0x000F → 0x001F). Status banner decodes statusword bits 10, 12, 13 live.
  • Zero encoder here — reads position_actual (0x6064), writes it into home_offset (0x607C), and updates the form field so Apply will persist it.

7.2 Motion profile tab

  • Profile velocity — 0x6081
  • Profile acceleration — 0x6083
  • Profile deceleration — 0x6084
  • Quickstop deceleration — 0x6085

These set the trapezoidal profile the drive uses when it moves under Profile Position / Profile Velocity modes (CiA 402 modes 1 / 3). Quick-stop decel is what the drive uses when E-STOP issues controlword 0x02.

7.3 Protection tab

  • Max following error — 0x6065 (counts); faults the drive if position error exceeds it.
  • Following error window — 0x6066 (ms); how long the error has to persist.
  • Position limit min / max — 0x607D:1,2; hard software travel limits.
  • Max motor speed — 0x6080 (counts/s).
  • Max torque — 0x6072 (‰ of rated).

Zero torque here reads torque_actual (0x6077) and writes it into torque_offset (0x60B2). Lets the current loading read as zero — useful when taring against gravity.

7.4 Encoder tab

  • Resolution — 0x608F:1 (counts per revolution).
  • Gear numerator / denominator — 0x6091:1,2.
  • Feed constant numerator / denominator — 0x6092:1,2.

7.5 Buttons

  • Read from drive — batch SDO upload over the full field set. Missing OD entries are reported in the log but don't abort the batch.
  • Apply — batch SDO download. Partial failures land in the log with per-field reason codes + a warning dialog.
  • OK — Apply + close. Cancel — discard.

8. Commissioning workflow

  1. Connect (Ctrl+K). Pick transport, set node range, Connect. Confirm the row in the table turns online = yes.
  2. Bring up. Select the row, open Control, click Bringup. State walks to Switched On. Click Enable. State: Operation Enabled.
  3. Configure. Drive → Configure Drive… → Read from drive. Review the Protection tab: pos limits and max torque should reflect the mechanical envelope. Apply.
  4. Home. On the Homing tab, either click Start Homing for a method-driven sweep, or drive the joint to a known position and click Zero encoder here to capture it as the origin.
  5. Tune. Open the Gains tab, Read the starting Kp/Ki. Use the Tuning dock: run a step response, read overshoot / rise time, adjust. For rotational dynamics, run a Bode sweep via the chirp waveform.
  6. Save. File → Save Profile writes the edited nameplate JSON. Save to Flash via Drive → Save Config to Flash (planned) persists runtime params on the drive side.
Always have an exit. Keep one hand near Space (E-STOP) the first time you enable a new drive. The diagnostic has no opinion about your mechanics — if limits aren't set right, motion can escape the safe envelope within a control cycle.

9. Tuning workflows

9.1 Step response

  1. Drive must be in Operation Enabled and in the target mode (e.g. Profile Position).
  2. Tuning dock → Step sub-tab. Pick target channel (position / velocity / torque), step size, offset, duration.
  3. Click Run step. The diagnostic overlays cmd vs. actual on the chart and computes overshoot (%), rise time (10 → 90 %), and steady-state error.
  4. Iterate: increase Kp until overshoot becomes visible, back off 20 %, then add Ki to kill the steady-state error.

9.2 Bode sweep

  1. Tuning dock → Bode sub-tab. Set frequency start / end (typically 1 Hz → 200 Hz) and sweep duration (30 s is generous).
  2. Run sweep. The diagnostic fires a chirp waveform; for each frequency the signal generator emits, a Goertzel correlator extracts in-phase and quadrature components of the measured output.
  3. Read magnitude crossover (−3 dB) and phase margin from the plot. Loop-shape accordingly.

10. PDO mapping — read, edit, apply

The diagnostic's PDO layer uses two hal_can endpoints — one for SDO, one for PDO. Keeping them separate avoids the single-consumer hal_can_set_cb from shadowing either traffic:

Editing the mapping happens via the PDO map dock:

  1. Click Edit… under TPDO1 or RPDO1.
  2. Add entries from the CiA 402 object catalog (Controlword, Statusword, Target / Actual Position, Velocity, Torque, Error Code, DC link voltage, …). Rearrange with Up / Down. Each entry contributes bit_length / 8 bytes to the frame.
  3. Click OK. The diagnostic runs the CiA 301 §7.2.2.2 remap dance via SDO:
Note. Against the in-tree simulator the PDO config objects (0x1600 / 0x1A00 and friends) are not exposed, so the remap aborts per-step with SDO abort code 0x06020000. The log surfaces the per-step reason. On a real CiA 402 drive the sequence completes and TPDO layout changes take effect on the next PDO emission.

11. Telemetry recording & export

  • Record telemetry (Ctrl+Shift+R) — toggled from the Data menu or toolbar. Currently a placeholder that writes to the log; CSV-per-snapshot is the planned evolution.
  • Export telemetry CSV… — dumps the currently buffered chart data (what the Telemetry pane has accumulated) to a timestamped CSV.
  • Clear charts — empties the scope buffer.

12. Emergency stop

The large red E-STOP button is pinned to the right of the toolbar and is always clickable. Shortcut Space is application-wide — it fires even when a setpoint spin box or name field has focus. On click:

  1. Any running signal generator is stopped.
  2. master_mgr_disable_all walks every slave into Switch-On Disabled (via controlword 0x00).
  3. The log records "E-STOP: disabled N slave(s)".

Re-enabling a slave after E-STOP is just the normal Bringup → Enable sequence on the Control tab.

14. Troubleshooting

SymptomLikely causeCheck
Configure Drive: "all fields aborted" Drive doesn't expose the CiA 402 config objects (simulator, minimal drives). Use a real drive, or leave the dialog for display only.
CO_FD_USDO_CLIENT: Session timeout Drive not responding. Wrong node id, bus down, transport mis-selected, or PDO callback eating SDO frames. Verify node id, transport; check the two UDP sockets came up.
PDO map Value column stays — No TPDO received — drive not emitting, COB-ID mismatch, or TPDO not enabled. Log should show "PDO online" once per slave; if not, sniff the bus.
E-STOP disabled Not connected yet — no slaves to disable. Connect first; it lights up automatically.
Start Homing does nothing visible Drive not in Operation Enabled before bit 4 pulse. Hit Enable on Control first.
Floating dock disappeared Window moved off-screen. Toggle it from the toolbar / View menu to re-show.

Appendix A — Transports in detail

A.1 UDP multicast (hal_can_udp)

The diagnostic opens two sockets per connection: one for SDO traffic, one for PDO. Both bind 0.0.0.0:23400 on the loopback interface with SO_REUSEADDR so they coexist in the same process. Each socket joins the 239.192.0.42 multicast group; IP_MULTICAST_LOOP stays on so other local sockets see our sends. Every frame rides a 12-byte header: 

[ sender_tag : 4 ][ cob_id : 4 ][ dlc : 1 ][ flags : 1 ][ rsvd : 2 ]

The sender tag filters self-echoes — each socket generates a random tag at create time.

A.2 ZLG USB-CANFD (hal_can_zlg)

libcontrolcanfd.so is opened via dlopen() at runtime; the shared object ships in src/backends/zlg/lib/ and is found through the RPATH. One endpoint per channel; PDO is not yet wired on this transport (V1 is SDO-only).
